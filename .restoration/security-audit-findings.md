# Sermon Works — Security Audit Findings

Running log of findings from the first-pass security audit (task #22 in the restoration backlog). Each confirmed high-severity issue is filed on the GitHub tracker; this file captures the broader landscape and the flags still awaiting manual verification.

## Methodology

Started from the PHPCS baseline (`.restoration/phpcs-baseline-*.txt`) run with the `WordPress-Extra` ruleset, which includes the `WordPress.Security.*` sniffs. Combined with manual code review of high-risk paths (the plugin bootstrap, REST API, import paths, bundled libraries). Bulk PHPCS signal is noisy — the audit distinguishes real findings from false positives by reading the affected code, not by trusting the flag count.

### REST API surface — audit outcome

`includes/class-sm-api.php` was the only plugin file hypothesised to own custom REST routes. On inspection it contains **no** `register_rest_route` calls — it is a hooks-only class that decorates the core CPT REST controller for `wpfc_sermon`. The only `register_rest_route` calls in the whole repo are inside `includes/vendor/CMB2/includes/rest-api/*` (covered by #15).

The `wpfc_sermon` CPT is registered with `show_in_rest => true`, `capability_type => 'wpfc_sermon'`, `map_meta_cap => true` (`includes/class-sm-post-types.php:265-314`), so REST insert/update is gated on `edit_wpfc_sermons` by the core controller. No missing-`permission_callback` finding was warranted.

The REST-surface vulnerabilities that *were* warranted (#16, #17, #18) are instead about the `rest_insert_wpfc_sermon` hook persisting REST-client-supplied params to post meta without sanitisation, whose values are then echoed unescaped in views. The three findings are the same class (URL-shaped meta written by `SM_API::save_custom_data()`, echoed into `href` / `src` / `download` / `data-*` attributes at multiple sinks) and a single `esc_url_raw()` pass at the source closes all three at once; per-sink output-escape changes provide defence-in-depth against any other writer reaching the same meta keys (CMB2 admin UI, `includes/admin/import/*` paths). Grep confirmed `rest_insert_wpfc_sermon` and `rest_prepare_wpfc_sermon` are the *only* non-vendor `rest_insert_*` / `rest_prepare_*` in the plugin, so this audit is complete for the REST-filter surface.

### Output-escape sweep — audit outcome

Full exhaustive triage of all 340 `WordPress.Security.EscapeOutput.OutputNotEscaped` flags from the WordPress-Extra ruleset. Priority was thoroughness (not sampling) because this sniff is the most XSS-relevant signal in the PHPCS output and the pre-WP.org window is when it's cheapest to find. Per-file methodology: read end-to-end, trace every echo to its data source, categorise as false-positive / admin-trusted defence-in-depth / true-positive stored or reflected XSS.

**Flag distribution (29 files, 340 total):** front-end view partials + widget + podcast feed = 203 flags (the high-risk surface); shortcode / template / main plugin = 29; admin-only render = 108; tests / core = 2. Biggest single files: `class-sm-export-sm.php` (60, admin-only XML download, all routed through a correct `wxr_cdata()` helper — false positives), `class-sm-admin-settings.php` (38, admin-only settings framework — false positives), `content-sermon-wrapper-start.php` (32, theme-compat `<div>` wrappers with class-attribute echoes of `apply_filters`-populated strings — false positives).

**Filed: #23, #24, #25, #26.** Two High stored XSS on the same REST-write family as #16-#18 (`bible_passage`, `sermon_description`) — both closed by the same `SM_API::save_custom_data()` per-key sanitiser pattern; `bible_passage` also flows through a direct `$wpdb` write at `sermons.php:232` that bypasses kses for `post_content`, folded into #23's remediation. One Medium stored XSS for `sermon_notes` (narrower — no REST writer) covering the URL-meta-in-`href` pattern. One Medium reflected XSS for three `$_SERVER['REQUEST_URI']` echoes on the admin import-export page.

**Subsumed by other issues** — not filed separately:
- Podcast feed CDATA break via `sermon_description` (`wpfc-podcast-feed.php:353-355`) → #24 covers it, including the `]]>` escape fix.
- `post_content` kses bypass via direct `$wpdb` write at `sermons.php:232` → #23 covers it in remediation step 3. Note if someone later sees value in filing this structural concern on its own — it affects any future meta value concatenated into the content builder.

**False-positive patterns confirmed benign:**
- **Theme-compat wrappers** (`content-sermon-wrapper-start.php`, `content-sermon-wrapper-end.php`): class-attribute echoes of `$additional_classes = implode( ' ', apply_filters( 'sm_templates_additional_classes', array(), $template ) );` — trust boundary is other plugin developers, no user-input flow. Uncode-theme branch echoes Uncode's own template helpers (`uncode_remove_p_tag`, `uncode_get_row_template`) — scoped to Uncode users.
- **Admin WXR export** (`class-sm-export-sm.php`): all content routed through the in-file `wxr_cdata()` helper at `:98-106`, which correctly escapes `]]>` → `]]]]><![CDATA[>`. Admin-trigger + Content-Disposition: attachment download + text/xml.
- **Admin settings framework** (`class-sm-admin-settings.php`, `html-admin-settings.php`, `class-sm-settings-*.php`): all values sourced from `get_option()` / `update_option()` — admin-set, admin-only page, admin-trusted.
- **WP core function returns** (`the_title()`, `get_the_ID()`, `get_post_class()`, `the_terms()`, `the_permalink()`, `the_date()`, `bloginfo_rss()`): admin-trusted via kses-on-save for non-`unfiltered_html` users. Missing escape is defence-in-depth, not a first-pass finding.
- **Translated strings** (`__()`, `esc_html_e()`, literal `echo` of hardcoded HTML): constant.
- **Shortcode attribute echoes** in `class-sm-shortcodes.php` and `content-sermon-filtering.php`: editor-level writer (who can place shortcodes), admin-trusted. Term-name in `title=""` attribute at `class-sm-shortcodes.php:305` flagged for defence-in-depth — missing `esc_attr` — but admin/editor-only writer path. Not filed.

**Low-priority observations, not filed:**

### UnsafePrintingFunction sweep — audit outcome

**Flag distribution (2 files, 20 total):**
- `includes/admin/views/html-admin-import-export.php` — 19 flags (lines 12, 15, 23, 59, 69, 74, 84, 93, 98, 102, 109, 110, 119, 127, 133, 160, 168, 174, 197). Headings, button labels, anchor text, card descriptions, footer note. All text contexts (`<h1>`, `<h3>`, `<p>`, anchor text, `<span>`), so `esc_html_e()` is the correct swap everywhere; `esc_attr_e()` was already being used correctly at the attribute-context sites in the same file.
- `sermons.php:647` — 1 flag. `_e( 'Import done!', ... )` inside an `admin_notices` closure that fires after the import handler finishes.

### GlobalVariablesOverride sweep — audit outcome

All 15 `WordPress.WP.GlobalVariablesOverride.Prohibited` flags reviewed. The sniff fires on any assignment to a name that matches a WordPress-conventional global (`$post`, `$post_type`, `$title`, `$link`, `$id`, `$order`, `$tax`, `$taxonomy`, `$cat_id`), regardless of whether the assignment is actually to a PHP global or to a local / loop / top-level-template scratch variable. Static analysis can't distinguish, so the burden is on manual review.

**Flag distribution (4 files, 15 total):**

- `includes/admin/export/class-sm-export-sm.php` — 2 flags (`$post` at :407, :442). `:442` is the canonical `foreach ( $posts as $post ) { setup_postdata( $post ); ... }` pattern — `setup_postdata()` requires `$post` to have been assigned; this is correct WP. `:407` iterates integer post-IDs in the WXR XML exporter; the file is an admin `die()`-after-emit script so no downstream code observes the `$post` state. Safe.
- `includes/sm-template-functions.php` — 2 flags (`$post` at :477, `$GLOBALS['post']` at :494). Both are inside `wpfc_sermon_single_v2()` which **deliberately swaps and restores** the `$post` global so the `content-sermon-single` partial renders against a caller-supplied sermon. The whole function is a save-and-restore wrapper; the flagged lines are doing exactly what the function advertises. Safe by design.
- `views/partials/content-sermon-wrapper-start.php` — 5 flags (`$post_type` ×3 at :159, :164, :326; `$tax` at :168; `$order` at :548). All inside the `case 'uncode':` branch (the Uncode-theme compat block, same branch as #32's POI sink at :376). The `$post_type` / `$tax` scratch variables build Uncode option keys like `_uncode_sermon_index_layout_width`; they mirror Uncode's own template conventions and renaming would decouple from the Uncode contract rather than improve safety. `$order` at :548 is a `foreach` array-key iterating the literal strings `'above'` / `'below'`, used only inside the loop body. No scope leak. Safe.
- `views/wpfc-podcast-feed.php` — 6 flags (`$id` ×2 at :49, :151; `$taxonomy` at :144; `$cat_id` at :214; `$title` at :244; `$link` at :245). Two are top-level template-scope scratch reads from `$settings['...']`. The podcast feed runs as a one-shot template_redirect handler that emits XML and `die()`s; no `the_loop()`-like downstream code observes any clobbered global. Safe.

**None filed.** Pattern matches the nonce-review false-positive clearance pattern from session 5 (`class-sm-shortcodes.php` 4/4, `class-sm-admin-settings.php` 10/10, warnings-only group 8/8). ### WPScan + Patchstack CVE sweep — audit outcome

Cross-reference of public CVEs against the `sermon-manager-for-wordpress` plugin slug, run post-bulk-cluster review. Three CVEs published, all affecting versions through 2.30.0 (the last upstream release), none with an upstream fix. Full technical details pulled from NVD / Patchstack / WPScan / Wordfence.

**CVE-2025-12368** (CVSS 6.4, Medium) — Authenticated (Contributor+) Stored XSS via `[sermon-views]` shortcode. **Filed as #35.** Sink located via NVD source pointer (`plugins.trac.wordpress.org` URL to `entry-views.php#L114`); confirmed by reading the code. `wpfc_entry_views_get()` concatenates `$attr['before']` and `$attr['after']` into the shortcode return with no escape. Researcher: zaim (via Wordfence).

**CVE-2025-63002** (CVSS 5.3, Medium, unauthenticated) — Missing Authorization. **Filed as #36 (suspected sink, ~85% confidence).** Patchstack did not disclose the sink; in-code audit identifies `wp_ajax_nopriv_wpfc_entry_views` → `wpfc_entry_views_update()` as the only candidate that matches the CVSS profile (unauthenticated state change, integrity-low, no scope change). Same file as #35 — the upstream `justintadlock/entry-views` library is broadly the vulnerable surface. Issue body carries a confidence caveat and a reproduction-then-reclassify note. Researcher: Legion Hunter (via Patchstack).

**CVE-2025-63000** (CVSS 6.4, Medium) — Authenticated (Contributor+) Stored XSS. **Filed as #37 (sink located, ~90% confidence).** No public database disclosed the sink location; in-code audit of `class-sm-shortcodes.php` (~1055 lines, 10 shortcode handlers) found two identical copy-pasted error-branch sinks at `:199` (`display_sermons_list()`) and `:459` (`display_images()`) that echo raw `$args['display']` into an `<em>` tag without escape. Reachable via any of `[list_sermons display='<script>…']`, `[sermon_images display='<script>…']`, or legacy aliases `tax=` / `taxonomy=` (compat merge at `:181-189`). Exhaustive grep for `$args[...]` / `$atts[...]` concatenation into HTML in the file returned four total sites; the two at `:199` / `:459` are the only vulnerable candidates (`:601` is allow-list validated; `:936` is `strtotime()` math). Same researcher as #12368, same pattern, same severity. Issue body carries a confidence caveat and a reproduction-then-reclassify note.

**Bundled `entry-views.php` status update.** Both #35 and #36 live in the bundled Justin Tadlock `entry-views` library, which — per #35's long-term remediation suggestion and the #28 taxonomy-images precedent — is a candidate for full removal in favour of core post-meta-based view counting. Worth a separate follow-up after the immediate XSS / auth fixes land. Not a filed issue yet; flagged here.

**No additional CVEs found** beyond the three WPScan entries. Patchstack's direct plugin-slug URL is a 404 (they surface Sermon Manager advisories only through CVE-specific pages). Wordfence returned no hits on `CVE-2025-63000` / `CVE-2025-63002` via search, but carries the CVE-2025-12368 advisory.
